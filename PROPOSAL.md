# さんごう防災3Dマップ — 品質向上・機能拡充 提案書

作成日: 2026-07-02

## 1. エグゼクティブサマリー

現状の「さんごう防災3Dマップ」は、PLATEAU 3D都市モデルを使った防災アプリとして
**技術的完成度は既に高い**（実行時データ取得＋3段フォールバック設計、単体テスト20件、
出典・ライセンス整備、モバイル最適化UI）。単なる「紙のハザードマップのデジタル化」ではなく、
建物単位のリスク判定・町全体統計・浸水の3D体感表示という独自機能まで実装済みで、
他自治体のPLATEAU活用事例と比べても遜色ない土台がある。

一方で、**三郷町に「画期的なアプリ」として提案し、実際に住民に使い続けてもらう**ためには、
以下3つの視点が不足している。

1. **災害時にこそ動く設計になっていない** — 現在は全機能が外部API（PLATEAU配信・地理院タイル）への
   リアルタイム接続を前提にしており、輻輳・停電・回線断が起きやすい災害時ほど使えなくなるリスクがある。
2. **「見る」で終わり「備える／動く」に繋がっていない** — リスクは分かるが、個人の避難行動計画
   （いつ・どこへ・何を持って逃げるか）に落とし込む機能がない。
3. **町が運用し、効果を説明できる体制がない** — 利用状況の把握、データの更新、住民からのフィードバック
   受け皿がなく、提案後に「作って終わり」になりやすい。

以下、公開前に対応すべき項目（P0）、他自治体との差別化になる目玉機能（P1）、
中長期の運用基盤（P2）に分けて具体的に提案する。

## 2. 現状評価（強み — そのまま提案の訴求ポイントにできる）

| 項目 | 内容 |
|---|---|
| ワンタップ診断 | 建物単位で浸水ランク・土砂災害警戒区域・最寄り避難場所（方角・距離・徒歩分数・経路）を即時表示 |
| 建物単位リスク判定 | PLATEAU CityGML属性から「3m以上浸水×2階建て以下＝垂直避難困難」を自動判定・色分け（`src/buildingrisk.js`） |
| 町全体統計ダッシュボード | 浸水想定面積・危険棟数を集計し、地域防災計画等の根拠データとして使える形で提示（`src/dashboard.js`） |
| 浸水の3D体感 | 地理院標高タイルとジオイド補正から実際の高さを持つ水柱を3D表示し、建物と見比べられる（`src/floodgrid.js`） |
| 堅牢なフォールバック設計 | 建物・避難所・地形・境界データすべてに複数系統のフォールバックとエラー表示があり、単一障害点を減らしている |
| 動作保証 | 座標変換・色判定・距離計算・実データ46件の整合性など、単体テスト20件が通過（`npm test`） |
| コンプライアンス | MIT License、PLATEAU/地理院タイル/CesiumJSの出典表記を整備済み |

## 3. 公開前に必須（P0）

### 3.1 実データでの検証（最優先）
`PROGRESS.md` に記録の通り、開発環境のネットワーク制約により**建物単位リスク判定が
三郷町の実モデルで動作するかは未検証**（浸水ランク属性の有無・名称はヒューリスティック検出）。
目玉機能が実機で動かないまま提案すると信頼を損なう。**実ブラウザでの動作確認を最優先で実施し、
属性が無い場合の代替表現（面積統計への誘導は実装済み）が適切に出るかを確認する。**

### 3.2 CIでのテストゲート
`.github/workflows/deploy.yml` は `npm run build` のみを実行し、**`npm test` を通さずにデプロイされる**。
PRごとにテストとビルドを検証するCIワークフローを追加し、壊れた変更が本番に出ないようにする。

### 3.3 プライバシーポリシー・個人情報の扱いの明文化
現在地情報を取得する機能があるが、取得したデータの扱い（サーバー送信なし・ブラウザ内処理のみ）を
明記したプライバシーページがない。README内のTODOのまま。自治体提案には必須。

### 3.4 OGP／検索対応
`index.html` にOGP（`og:image`等）・Twitter Cardタグがなく、SNS共有時にサムネイルが出ない。
災害時・避難訓練時にSNSでリンクが拡散されることを想定すると重要。

### 3.5 やさしい日本語・最低限のアクセシビリティ監査
高齢者・外国人住民を含む全住民が対象の防災アプリとして、スクリーンリーダーでの読み上げ確認、
文字サイズ拡大時のレイアウト崩れ確認、色覚多様性を考慮した凡例配色の再確認を行う。

## 4. 「画期的」と言える差別化機能（P1）

### 4.1 オフライン対応（PWA化）— 最重要
**災害時ほど通信が繋がらないのに、現状は全機能がオンライン前提**という構造的な弱点がある。
Service Workerでアプリ本体（HTML/CSS/JS）と同梱避難所データ・凡例・診断済み結果をキャッシュし、
「電波が無くても、最後に確認した自宅周辺のリスクと避難所は見られる」状態を作る。
ホーム画面への追加にも対応させ、平時からインストールしてもらう動線を作る。
→ 他自治体のPLATEAU防災アプリとの明確な差別化ポイントであり、町への提案の目玉にできる。

### 4.2 マイ・タイムライン機能
国交省が推進する「マイ・タイムライン」（自分の避難行動計画）の考え方を、
既存の地点診断結果と接続して実装する。世帯構成（高齢者・乳幼児・要配慮者の有無）を入力すると、
警戒レベルに応じた行動チェックリストと避難開始の目安時刻を提示し、端末に保存できる。
既に持っている「浸水深・最寄り避難所・徒歩分数」のデータをそのまま活用でき、追加の外部データ取得は不要。

### 4.3 リアルタイム防災情報の重畳
気象庁の警報・注意報（大雨警報等）や、大和川・竜田川の水位情報（国交省 川の防災情報）を
定期取得して地図上に表示する。現状のハザード表示は「想定」の静的情報のみで、
「今まさにどれくらい危険か」が分からない。これを足すことで平時の閲覧アプリから
災害時に開く実用アプリへ引き上げられる。

### 4.4 診断結果の音声読み上げ・共有
Web Speech APIで診断結果を読み上げる機能（視覚障害者・高齢者向け）と、
診断結果をQRコード化して家族と共有・印刷できる機能を追加する。
紙のハザードマップ配布時に「QRを読むとその場所の詳細診断が見られる」導線としても使える。

### 4.5 避難ルートのハザード回避
現在の経路リンクはGoogleマップの直線的な徒歩ルートで、浸水区域を突っ切る可能性がある。
自前の詳細ルーティングは工数が大きいため、まずは「経路が警戒区域を通過する場合は警告を表示する」
簡易チェックから着手する。

## 5. 運用の持続可能性（P2）

- **町向け管理画面／データ更新**: 避難所情報などを町職員がコードを触らず更新できる仕組み
  （Googleスプレッドシート連携やヘッドレスCMSなど、軽量な方式で十分）
- **利用状況の可視化**: プライバシーに配慮した形（個人を特定しない集計）で、
  診断実行数・よく見られる地域などを把握できるようにし、町への効果説明・予算確保根拠に使う
- **住民フィードバック導線**: アプリ内から誤り報告・改善要望を送れる簡易フォームを設置
- **多言語対応**: 英語・やさしい日本語への切替（外国人住民・観光客対応）

## 7. まとめ

現状のアプリは技術基盤として優秀であり、「作り直す」必要はない。
**P0（信頼性・公開準備）を固めた上で、P1のうち特にPWAオフライン対応とマイ・タイムライン機能**を
実装できれば、「見せるハザードマップ」から「災害時に本当に使える・平時から備えさせるアプリ」へと
性質が変わり、三郷町に対して他自治体にない提案価値を打ち出せる。
