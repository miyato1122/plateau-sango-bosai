# さんごう防災3Dマップ — 三郷町提案に向けた品質向上提案書

作成日: 2026-07-02
対象: 奈良県三郷町（PLATEAU 2025・市区町村コード 29343）
前提: 本アプリは既に「タップ診断／3D建物／ハザード重畳／避難場所／建物別リスク／統計ダッシュボード／浸水3D体感」まで実装済み。本書はこれを **自治体が公式採用できる品質＝「画期的な防災アプリ」** へ引き上げるための提案。

---

## 1. 現状評価（率直な棚卸し）

### すでに優れている点
- **狙いが明確**: 「タップするだけで自分の家のリスクと逃げ先がわかる」= 紙のハザードマップの弱点（自分ごと化しにくい）を的確に突いている。
- **配信API直読み設計**: 大容量データを同梱せず、PLATEAU配信・地理院タイルを実行時取得。GitHub Pagesで動く軽量な静的サイト。
- **多段フォールバック**: 避難所（API→同梱→地理院）、地形（PLATEAU-Terrain→地理院DEM）と堅牢。
- **ライセンス・出典の整備**: R3JHs 778 の帰属表記、CC BY、Cesium の Apache 2.0 まで丁寧。自治体提案で重要な"コンプラ耐性"がある。
- **純粋ロジックの単体テスト20件**: 座標変換・色判定・距離方位・実データ整合性まで検証。プロトタイプとしては良質。

### 「自治体公式アプリ」としてはまだ足りない点（本書で解決する）
1. **災害時に動かない可能性がある**（最重要）— 完全オンライン依存。実災害時は通信が輻輳・途絶する。オフラインで動かない防災アプリは本番で機能しない。
2. **避難経路が危険側に出うる** — 経路リンクが Google マップの通常徒歩ルート。**浸水域・土砂警戒区域の中を通る経路をそのまま提示しかねない**（`src/main.js` の `route-link`）。防災アプリとして致命的。
3. **平常時の計画ツールに留まる** — 気象警報・河川水位などの**リアルタイム情報がなく**、「いま逃げるべきか」に答えられない。
4. **要配慮者・多言語・アクセシビリティ未対応** — 三郷町は高齢化が進む。JIS X 8341（ウェブアクセシビリティ）・やさしい日本語・外国人住民対応は自治体調達の実質必須要件。
5. **データの正確性が"タイル色読取り"依存** — 賢い実装だが、自治体が責任を持って公開する情報としては、公式ハザードマップとの照合・出典の厳密性が不足。
6. **運用・保守の受け皿がない** — 職員がコードを触らずにデータ更新・お知らせ配信できる仕組み、利用統計、監視、個人情報保護方針が未整備。議会・住民への説明責任を果たせない。
7. **エンジニアリング品質** — TypeScript未導入、CIテストゲート・E2E・エラー監視なし、`main.js` が505行に肥大化。中長期の保守に耐える構造への転換が必要。

## 2. 「画期的」にする5つの柱

「機能を足す」提案ではなく、**自治体が公式に採用でき、住民の命を守れる**水準へ引き上げる。差別化の核は次の5点。

### 柱A. 災害時にも動く — PWA・オフライン対応（最優先）
- **Service Worker + アプリマニフェスト**でホーム画面設置・オフライン起動を実現。
- 三郷町域の**ハザードタイル・避難所・淡色地図・地形**を事前キャッシュ（町域だけなら数十MBに収まる。全国依存を切る）。
- 診断ロジック（`risk.js` の色判定）はローカル完結なので、**キャッシュ済みタイルがあればオフラインでも診断可能**にできる。
- 「いま電波がなくても、あなたの家のリスクと避難場所は出せる」= 防災アプリとして本質的な価値。

### 柱B. 安全な避難経路 — ハザード回避ルーティング
- 現状の「Googleマップに丸投げ」を廃し、**浸水域・土砂警戒区域を避ける経路探索**へ。
  - 町域の道路ネットワーク（PLATEAU道路／OSM）にハザードタイル判定を重ね、危険セルを高コスト化した最短経路。
  - 3D上に**安全な避難経路を色付きラインで描画**（浸水を通る箇所は警告表示）。
- **逃げ地図（避難時間マップ）**: 町内各地点から最寄り避難所までの徒歩到達時間を面で色分け。「避難に15分以上かかる空白地帯」を可視化 → 地域防災計画の根拠になる。PLATEAUの3Dと極めて相性が良く、**議会・住民説明で"効く"ビジュアル**。

### 柱C. リアルタイム防災 — 気象・河川・警報連携
- **気象庁 防災情報 / キキクル（危険度分布）**、**川の防災情報（大和川・竜田川の水位・氾濫危険水位）** を取り込み、平常時の計画ツールから**「いま逃げるべきか」を判断できるライブツール**へ。
- 危険度に応じて**警戒レベル（1〜5）を画面上部に常時表示**。マイ・タイムライン（後述）と連動。
- （発展）Web Push による**プッシュ通知**。町の避難情報と連動できれば、住民が"開かなくても"警告を受け取れる。

### 柱D. 一人ひとりの避難計画 — マイ・タイムライン + 要配慮者支援
- 国交省が推進する**マイ・タイムライン**をデジタル化。自宅を登録すると、その場所のリスクに応じて「台風◯日前に何をするか」の行動計画を自動生成・保存（ローカル保存＝個人情報を送信しない）。
- **要配慮者モード**: 徒歩速度・車椅子・冠水回避を前提にした経路・所要時間、福祉避難所の明示。三郷町の高齢化に直結する価値。
- **やさしい日本語 / 英語**切替、**音声読み上げ・大きな文字・高コントラスト**（WCAG 2.1 AA / JIS X 8341-3 準拠）。

### 柱E. 自治体が運用できる — 保守・ガバナンス・説明責任
- **職員向けデータ更新フロー**: 避難所・お知らせを職員がGit不要で更新できる仕組み（GeoJSON差し替え手順のドキュメント化、将来的には簡易管理画面）。データ検証（CRC/スキーマ）を自動化。
- **匿名利用統計**: 個人を特定しない利用数・診断数の集計。議会・住民への効果報告に使える（プライバシー配慮の設計）。
- **プライバシーポリシー / 個人情報保護 / 免責・アクセシビリティ方針**の整備。位置情報は端末内処理でサーバー送信しない旨を明記。
- **公式ハザードマップとの整合確認**プロセスと、データ出典・更新日の画面表示。

## 3. エンジニアリング品質の引き上げ

自治体が数年単位で保守・委託できる土台にする。

| 領域 | 現状 | 提案 |
|---|---|---|
| 型安全 | Vanilla JS | **TypeScript** 段階導入（まず `lib/` の純粋ロジックから） |
| テスト | 単体20件 | **CIでテストゲート**（PRで自動実行）、**Playwright によるE2E**（タップ診断・避難所表示の実ブラウザ検証） |
| 監視 | なし | **エラートラッキング**（Sentry等）で本番の失敗を検知 |
| 構造 | `main.js` 505行 | UI/状態/副作用を**モジュール分割**、イベント配線を整理 |
| パフォーマンス | 都度タイル取得 | タイルキャッシュ戦略、3D Tilesの描画最適化、初回表示の体感速度改善 |
| アクセシビリティ | 一部のみ | 自動チェック（axe）をCIに組込み、キーボード操作・フォーカス管理・ARIA整備 |
| セキュリティ | — | CSP設定、依存の脆弱性スキャン、公開トークンの取扱い方針明記 |

## 4. 段階的ロードマップ

「一度に全部」ではなく、**各フェーズ単体で三郷町に見せられる**刻み方にする。

### フェーズ1（〜2週間目安）— "公式提案の第一稿"を作る
- **PWA化 + 町域オフラインキャッシュ**（柱A）
- **ハザード回避の警告つき経路 / 逃げ地図の簡易版**（柱B の入口）
- CIテストゲート + E2Eの骨組み、`main.js` の分割着手
- プライバシーポリシー・免責・出典更新日の整備
→ この時点で「災害時に動く」「安全な逃げ先を示す」= 既存プロトタイプから**質的に別物**になる。

### フェーズ2 — "住民に届く"
- **リアルタイム気象・河川・キキクル連携 + 警戒レベル表示**（柱C）
- **やさしい日本語 / 英語、アクセシビリティ AA 準拠**（柱D 一部）
- 匿名利用統計・管理運用ドキュメント（柱E）

### フェーズ3 — "画期的"の完成
- **マイ・タイムライン + 要配慮者モード**（柱D）
- Web Push 通知、逃げ地図の本格版（3D到達時間面）
- 職員向け更新の仕組み、TypeScript移行の完了

## 5. すぐ着手できる第一歩（本ブランチで実装可能）

現環境（サンドボックス）は外部タイルに接続できないためブラウザ実測はできませんが、以下は**コードとして安全に実装・テストまで到達**できます。まずここから着手することを推奨します。

1. **PWA基盤**: `manifest.webmanifest` + Service Worker（アプリシェル + 地理院/ハザードタイルの stale-while-revalidate キャッシュ）。オフライン起動を実現。
2. **経路の安全化（第一段）**: 避難所への直線ではなく、経路上の浸水域を診断ロジックで判定し、**「この経路は浸水域を通ります」警告**を診断カードに追加。Googleマップ任せの危険を最初に潰す。
3. **CIテストゲート**: `.github/workflows` に `npm test` を追加し、PRで自動実行。品質の後退を防ぐ土台。
4. **`main.js` の分割**: UI配線・診断・レイヤ管理を分離し、以降の機能追加に耐える構造へ。

## まとめ

現状は「よくできた個人開発プロトタイプ」。**画期的な自治体公式アプリ**へ変えるための決定打は、機能の数ではなく次の3つです。

1. **災害時に動く**（PWA・オフライン）
2. **安全な避難行動に直結する**（ハザード回避経路・逃げ地図・リアルタイム警戒）
3. **自治体が責任を持って運用できる**（アクセシビリティ・多言語・保守・ガバナンス）

この3点を満たせば、「PLATEAUを使った"見せる"3Dマップ」から、「**三郷町が住民の命を守るために公式配布する防災インフラ**」へと格が変わります。
